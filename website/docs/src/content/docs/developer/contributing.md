Thanks for your interest in Waffled. This guide covers how to get set up, run the
checks, and open a good PR. When in doubt, match the surrounding code.

## Local setup

Waffled runs as a Docker Compose stack driven by the root `./waffled` bash CLI.
Requires **Docker** and **Node 24** for source development. The production API
container remains on Node 20.

```bash
git clone <repo-url> waffled
cd waffled
./waffled up          # first run bootstraps infra/compose/.env with generated secrets
./waffled web         # Vite HMR for apps/web
```

Other useful CLI verbs: `./waffled down`, `rebuild`, `migrate`, `backup`, `restore`,
`doctor`, `admin`, `logs`, `status`. See [Local development](/developer/local-development/)
for the full workflow.

## Project layout

- `apps/api` ‚Äî backend (TypeScript, lambda-api, bundled with esbuild; Postgres
  migrations via node-pg-migrate). Its own `package.json` / `package-lock.json`.
- `apps/web` ‚Äî kiosk / web app (React + Vite). Its own `package.json`.
- `apps/ios` ‚Äî SwiftUI app (Xcode project generated by XcodeGen from `project.yml` ‚Äî
  run `xcodegen generate` after adding files). See [iOS development](/developer/ios/).
- `infra/compose` ‚Äî Docker Compose stack (postgres, api, powersync, caddy,
  backup, optional observability).
- `website` ‚Äî this documentation site (Astro Starlight).
- `docs` ‚Äî project documentation.

This is **not** an npm-workspaces monorepo ‚Äî `apps/api` and `apps/web` each
manage their own dependencies with `npm`.

## Running tests

Run the checks for the relevant app before opening a PR ‚Äî CI runs all of it.

```bash
# API ‚Äî vitest + Testcontainers integration (needs Docker running)
cd apps/api && npm test
cd apps/api && npm run typecheck

# Web ‚Äî vitest / jsdom
cd apps/web && npm test
cd apps/web && npm run typecheck
```

For iOS, run the Swift Testing suite:

```bash
xcodebuild test -project Waffled.xcodeproj -scheme Waffled \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro'
```

## Commits

Use conventional-commits-style messages with a **scope**:

```
feat(meals): add recipe scheduling
fix(chores): correct approval ordering
docs: expand contributing guide
ci: cache node_modules
```

## Pull requests

- **Keep PRs focused** ‚Äî one logical change per PR is much easier to review.
- **CI must pass** (tests + typecheck for both apps).
- **Add tests** for behavior changes ‚Äî API changes get integration tests, web
  changes get component / unit tests.
- **Update `CHANGELOG.md`** ‚Äî add an entry under the `[Unreleased]` section.
- **Update docs** when you change behavior or config.

## Questions & security

Open a **GitHub Issue** for bugs and feature requests, or start a **GitHub
Discussion** for questions and ideas. For **security issues**, follow
`SECURITY.md` and report privately ‚Äî **not** as a public issue. All contributors
are expected to follow the `CODE_OF_CONDUCT.md`.

## CI/CD

- **`.github/workflows/ci.yml`** runs on PRs and on push to `main`. API tests
  run on Node 20 to match production; web checks run on Node 24 to match the
  browser toolchain. Separate jobs cover migration hygiene, CLI tests,
  typechecking, web tests, API integration tests, and API Docker E2E.
- **`.github/workflows/publish-images.yml`** runs **only on `v*` tags** (or
  manual dispatch). It builds three multi-arch (amd64 + arm64) GHCR images ‚Äî
  `waffled-api`, `waffled-caddy` (which bakes the web SPA), and `waffled-backup`
  ‚Äî and cuts a GitHub Release.
- **iOS is Xcode Cloud**, separate from the above (see
  [iOS development](/developer/ios/)).

## Release process

`CHANGELOG.md` follows [Keep a Changelog](https://keepachangelog.com/) and
[SemVer](https://semver.org/). To cut a release:

1. Move the notes from `[Unreleased]` into a new version section in
   `CHANGELOG.md`.
2. Tag and push:

   ```bash
   git tag vX.Y.Z
   git push origin vX.Y.Z
   ```

The `v*` tag triggers `publish-images.yml`, which builds the GHCR images and
cuts the GitHub Release.
